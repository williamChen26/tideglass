# Architecture

## 当前状态

这个仓库目前是一个智能体优先的工程骨架，还没有绑定到具体业务代码或单一技术栈。

它的目标是先建立一套能够支持长期协作的记录系统、计划系统和最小治理机制，再逐步长出产品代码。

## 设计目标

1. 让新加入的智能体能在几分钟内找到正确的上下文。
2. 让产品、架构、计划和质量约束都可以在仓库内被发现。
3. 让反复出现的约束逐步从“文档说明”升级为“自动检查”。

## 推荐结构

在开始落业务代码后，建议按显式边界组织：

```text
src/
  domains/
    <domain>/
      types/
      config/
      repo/
      service/
      runtime/
      ui/
  providers/
  shared/
```

## 约束建议

1. `types -> config -> repo -> service -> runtime -> ui` 只允许单向依赖。
2. 横切能力通过 `providers/` 或其他显式边界接入。
3. 共享工具放入 `shared/`，避免每个模块重复发明辅助函数。
4. 边界输入输出要做显式校验，不依赖“猜测的数据形状”。
5. 当文档约束被频繁违反时，应优先把它转成 lint、测试或脚本。

## 文档即控制面

这个仓库把以下内容视为一等工件：

- 产品规格
- 执行计划
- 设计决策
- 技术债
- 安全、可靠性、质量约束

## 演进方式

1. 先在 `docs/product-specs/` 定义要解决的问题。
2. 再在 `docs/exec-plans/active/` 写如何实现。
3. 实现后把偏好与经验沉淀回文档或检查脚本。
4. 当模式稳定时，再抽成模板、命令或 CI。

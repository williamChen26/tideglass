# Quality Score

质量不是一句"看起来不错"，而是持续维护的可见分数。

## 评分维度

| 维度 | 说明 | 当前评分 |
|------|------|----------|
| 架构清晰度 | 层级分明、依赖单向、边界显式 | — |
| 文档新鲜度 | 文档与代码一致、索引完整、无过时内容 | — |
| 测试覆盖 | 单元+集成+E2E 覆盖率 ≥ 80% | — |
| 可靠性 | 启动恢复、性能预算、降级策略 | — |
| 安全性 | 信任边界、输入校验、凭证管理 | — |
| 智能体可读性 | 智能体能否从仓库本身获取足够上下文 | — |

评分标准: A (优秀) / B (良好) / C (需改进) / D (风险)

## Evaluator 评审标准

Harness 系统中 Evaluator 的评判维度直接映射到此处：

### 合约验证 (硬门控)

每条验收标准必须独立验证，单条失败 = 整轮被拒。

| 检查项 | 通过条件 |
|--------|----------|
| 功能正确性 | 验收标准中每条 AC 有对应测试且通过 |
| 测试覆盖 | 新增代码覆盖率 ≥ 80% |
| 构建通过 | 无编译错误、无类型错误、无 lint 错误 |
| 文件规范 | 文件 < 800 行，函数 < 50 行 |
| 不可变模式 | 无就地修改，创建新对象 |

### 质量观察 (非阻断)

Evaluator 同时记录不影响判定的质量观察，沉淀为改进建议：

- 命名清晰度
- 抽象层次是否恰当
- 是否存在重复模式可提取
- 错误处理是否充分
- 边界校验是否完整

## 评分节奏

- 每次 Sprint 通过后，Evaluator 的 evaluation.md 自动贡献质量数据
- 每周人工审视一次评分表，更新趋势
- 当某维度降至 D 级时，优先修复

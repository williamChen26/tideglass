# Security

安全约束应前置到设计与实现边界，而不是事后补救。

## 信任边界

```
用户输入 ──→ [验证层] ──→ 业务逻辑 ──→ [输出编码] ──→ 渲染/响应
外部 API  ──→ [解析+校验] ──→ 内部类型
文件上传  ──→ [类型检查+大小限制] ──→ 存储
```

**原则**: 在边界处解析和验证，内部代码信任已验证的类型。

## 必检清单

每次提交前（Generator 自检 + Evaluator 审查）：

- [ ] 无硬编码密钥（API key, password, token）
- [ ] 所有用户输入已验证
- [ ] SQL 查询使用参数化（无字符串拼接）
- [ ] HTML 输出已转义（防 XSS）
- [ ] CSRF 保护已启用
- [ ] 认证/授权逻辑已验证
- [ ] 错误消息不泄露内部状态
- [ ] 敏感操作有审计日志

## 凭证管理

- 永远不在源码中硬编码密钥
- 使用环境变量或密钥管理服务
- 启动时验证必需密钥是否存在
- 泄露后立即轮换

## Harness 安全考量

智能体系统引入额外安全面：

| 风险 | 缓解 |
|------|------|
| Agent 执行任意命令 | Generator 限制在沙箱内，Evaluator 不可修改代码 |
| 提示注入 | Agent 只读取仓库内文件，不处理未信任的外部输入 |
| 状态篡改 | meta.json 由编排器管理，agent 不直接修改他人输出 |
| 密钥泄露到 harness 日志 | 运行状态在 docs/exec-plans/ 中版本化管理 |

## 敏感操作审计

待业务代码落地后补充具体审计点。

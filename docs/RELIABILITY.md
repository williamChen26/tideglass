# Reliability

可靠性约束需要在仓库内可见、可测、可迭代。

## 可靠性预算

| 维度 | 目标 | 度量方式 |
|------|------|----------|
| 启动时间 | < 3s (开发), < 1s (生产) | 计时脚本 |
| 关键路径延迟 | < 2s (用户可感知操作) | E2E 测试断言 |
| 错误恢复 | 所有可恢复错误有降级路径 | 集成测试 |
| 数据完整性 | 写入操作原子化，不留半成品 | 事务测试 |

## 失败场景与降级策略

待业务代码落地后补充具体场景。模板：

| 场景 | 影响 | 降级策略 | 恢复方式 |
|------|------|----------|----------|
| 外部 API 不可达 | 功能 X 不可用 | 显示缓存数据 + 错误提示 | 自动重试 + 断路器 |
| 数据库连接失败 | 全站不可用 | 只读模式 / 维护页面 | 连接池重建 |

## Harness 运行的可靠性

Harness 系统本身也需要可靠性保证：

| 场景 | 处理方式 |
|------|----------|
| Agent 未产出必需文件 | 重试一次，然后升级到人类 |
| 合同审查被拒 | Generator 修改合同，重新提交 |
| 测试基础设施损坏 | Generator 必须先修测试再继续 |
| 运行中断（上下文耗尽） | 从 meta.json 恢复到当前 sprint |
| Sprint 3 轮仍然失败 | 跳过该功能或升级到人类 |
